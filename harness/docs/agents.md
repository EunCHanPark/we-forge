# Agent Rules — we-forge

> **이 문서는 Claude Code 등 에이전트가 읽는다.**
> 사람용 원칙 설명은 [`harness/core/docs/core-beliefs.md`](../../../../harness/core/docs/core-beliefs.md) 참조.
> 상위 권위 문서는 [`harness/core/docs/constitution.md`](../../../../harness/core/docs/constitution.md).
>
> 각 규칙은 `R<카테고리>.<번호>` 식별자와 원칙 근거(`B<n>`)를 갖는다.
> 규칙 추가/수정은 사용자만 한다. **에이전트는 이 파일을 자율 수정하지 않는다.** (R7.1, Constitution NN5)

---

## 읽는 방식

작업 시작 전 해당 카테고리의 규칙을 확인한다. 규칙 형식:

```
### R<id> — <제목>                              [근거: B<n>]
트리거: 언제 이 규칙을 확인하는가
행동: 무엇을 하는가 / 하지 않는가
검증: 자기 검증 방법 (grep/명령어/체크리스트)
위반 시: 어떻게 대응하는가
```

**우선순위**: Constitution §4 Principle Priority를 따른다 (데이터 > 정확성 > 실천).
구체적으로: R10.x(안전) > R9.x(복구) > R8.x(관측) > R2.x(진실성) > R11.x(문서 선행) > R13.x(Agent Teams) > R1.x/R6.x(경계) > R4.x(테스트) > R12.x(PR 분할) > 기타.

## R0 — 메타 규칙

에이전트 행동 일반 원칙. 모든 세션에 적용.

### R0.1 — 불확실하면 물어본다                           [근거: 전반]
**트리거**: 사용자 요청이 모호하거나, 가정 없이 진행할 수 없는 결정 지점에 도달했을 때.
**행동**: 작업을 중단하고 명시적 질문. 추측으로 코드 생성 금지.
**질문 판단 기준**:
- 가정이 2개 이상 분기되고 선택이 되돌리기 어려우면 → 물어본다
- 가정이 1개이고 되돌리기 쉬우면 → 가정을 명시적으로 말하고 진행
**금지**: "아마도 이럴 겁니다" 식의 침묵 가정.

### R0.2 — 요청 범위를 넘지 않는다                       [근거: 전반, B12]
**트리거**: 작업 중 "관련 있어 보이는" 다른 파일/이슈를 발견했을 때.
**행동**:
- 요청 범위 안의 변경만 수행
- 범위 밖 발견은 `# TODO(사용자 확인):` 주석 또는 별도 보고로 처리
- 파일 삭제/이동/대규모 리네임은 명시적 허가 필수
**금지**: "김에 같이 고쳤습니다" 류의 범위 확장.
**검증**: PR diff가 요청 내용과 1:1 대응되는지 자기 점검.

### R0.3 — 테스트 실패는 수정, 삭제가 아니다              [근거: B4, NN9]
**트리거**: 테스트 실패에 직면했을 때.
**행동**:
- 실패 원인을 먼저 설명 (코드 버그인가, 테스트 버그인가, 환경 문제인가)
- 코드 버그면 코드 수정
- 테스트 버그면 테스트 수정 후 *이전 버전이 왜 틀렸는지* 명시
- 판단 불가면 R0.1로 대기
**금지**: 테스트 삭제, `skip`/`xfail` 남발, `assert True`로 교체.
**검증**: 수정된 테스트가 의도한 동작을 실제로 검증하는가.

### R0.4 — 맥락이 오래되면 재확인                         [근거: B7]
**트리거**: 세션 초반 읽은 파일의 내용을 5턴 이상 지나 수정할 때. 또는 브랜치가 변경됐을 때.
**행동**: 수정 전 `view`로 현재 상태 재확인. 캐시된 기억으로 str_replace 하지 않는다.
**검증**: str_replace 직전의 마지막 view 타임스탬프 확인.

### R0.5 — 실질적 작업 전 advisor 호출                    [근거: B7]
**트리거**: 다음 중 하나에 해당할 때.
- 코드 작성/수정을 시작하려 함 (orientation/탐색 제외)
- 커밋/PR/머지 등 외부 상태를 바꾸는 결정을 내리려 함
- 플랜 작성·아키텍처 선택 등 되돌리기 어려운 판단을 하려 함
- 접근 방식을 바꾸려 함
- "작업 완료" 선언 직전
**행동**: 상위 리뷰어(advisor)에게 전체 맥락을 전달하여 접근 방향·스코프·리스크를 검증 받는다. 사용 도구는 플랫폼에 따라 다름(예: Claude Code의 `advisor()` 콜).
**예외**: 파일 탐색·문서 조회·`git status` 같은 read-only orientation 단계.
**금지**: orientation 이후 첫 substantive 변경을 advisor 없이 진행.
**검증**: 2-stage review 파이프라인의 시작 지점 — 상세는 `harness/core/workflows/pipeline.md` §6 "2-stage Review 파이프라인" 참조.
**위반 시**: advisor 호출 후 재시작. 이미 작업한 산출물은 유지하되 advisor 판결에 맞춰 수정한다.

### R0.6 — 병렬 작업 선택 트리                            [근거: B3, B12]
**트리거**: 작업을 여러 워커에 나눠 병렬로 실행하려는 결정 지점.
**행동**: 작업 성격에 따라 아래 3가지 메커니즘 중 **하나**를 선택한다. 섞어 쓰지 않는다.

| 상황 | 선택 | 도구 예시 |
|------|------|----------|
| 코드베이스 탐색·파일 찾기·키워드 검색 | **탐색 전용 서브에이전트** | Claude Code: `Agent(subagent_type=Explore, ...)` |
| 독립적인 서브태스크를 병렬 실행 (워커 간 통신 불필요, 결과만 받음) | **Task 병렬 호출** | Claude Code: 단일 메시지에 `Task(...)` multi-tool |
| 워커끼리 토론·교차검증·공유 task list·상호 메시징 필요 | **Agent Team** (플랫폼 native) | Claude Code: `TeamCreate` + teammates (상세: R13) |

**판단 기준**:
- "결과만 받으면 됨" → Task / Explore
- "워커끼리 서로 이야기하거나 합의해야 함" → Agent Team
- "파일/심볼 찾기" → Explore 전용 (경량)

**금지**: Agent Team이 필요한 상황(경쟁 가설, cross-layer, PR 멀티렌즈)에 `Task` multi-tool을 억지로 쓰는 것.
**검증**: 작업 시작 전 상황을 3개 중 어디에 매핑했는지 스스로 명시. advisor에 전달 시 이 매핑도 포함.
**위반 시**: 중간에 "Team이 필요했음"을 깨달으면 현 작업 중단 → advisor 재호출 → Team 재설계.

## R2 — 레포 진실성

### R2.1 — 서버 수동 변경은 레포로 되돌아간다              [근거: B2]
**트리거**: 프로덕션 서버 구성/리소스를 직접 수정해야 하는 상황.
**행동**: 24시간 내 레포에 반영 PR 작성. PR 제목에 `[hotfix-reconcile]`.

**행동**: 코드 변경 전 `.env.example`에 키와 한 줄 설명 추가.

**트리거**: `secrets/` 리소스 작업.
**금지**: 실제 값을 커밋하거나 자동 apply.
**검증**: 커밋 전 `git diff --cached -- secrets/` 스캔.

## R7 — 하네스 자기 수정

### R7.1 — 하네스 파일 자율 수정 금지                     [근거: NN5]
**트리거**: `harness/core/`, `harness/docs/agents.md`, `constitution.md` 등 수정 필요.
**행동**: 사용자에게 변경 제안만. 승인 후에만 수정.

### R7.2 — 반복 실수는 원인 층위를 분류한다              [근거: B7]
**트리거**: 같은 세션 또는 과거 세션에서 동일 실수 반복 감지.
**행동**: 원인을 다음 중 하나로 분류하여 사용자에게 보고:
- (a) **하네스 공백** → 규칙 초안 제안 (R7.1에 따라 사용자 반영)
- (b) **코드 구조 문제** → 리팩터링 제안 (명명, 모듈 분리 등)
- (c) **작업 범위 과다** → 작업 분할 제안 (R12.x 참조)
**금지**: 자동으로 (a)로 단정하여 하네스에 규칙을 늘리는 것. 코드 구조가 원인이면 코드를 고쳐야 한다.
**기록**: 판단 과정과 결과를 평가 기록(예: `harness/plans/evaluation-log.md`)으로 제시.
**검증**: 분류 결과와 근거를 평가 기록 초안에 포함했는지 확인. (a)/(b)/(c) 중 하나로 단정하지 말고 후보를 병기했는지 점검.
**위반 시**: 자동으로 (a)로 단정한 하네스 추가 제안은 철회. 코드 구조 가설을 다시 수립하고 사용자에게 재보고.

## R8 — 관측성

### R8.1 — 핵심 경로는 로그 + trace_id                   [근거: B8]
**트리거**: 새 엔드포인트/백그라운드 작업 추가.
**행동**: 구조화 로그 + trace_id 전파. `except: pass` 금지.

### R8.2 — 핵심 경로는 구조화 로그 필수                  [근거: B8]
**트리거**: 핵심 비즈니스 흐름(인증, 외부 시스템 연동, 주요 트랜잭션 등) 코드 작성.
**행동**: JSON 로그, 최소 필드: `timestamp`, `level`, `service`, `trace_id`, `event`.
**금지**: `print()`, `logging.info("something happened")` 식의 비구조 로그 (디버그 전용 제외).
**검증**: 주요 경로 로그 샘플(JSON 1줄)에 `timestamp/level/service/trace_id/event` 필드가 모두 있는지 점검. CI 또는 수동 grep으로 `print(` 재등장 없음 확인.
**위반 시**: 비구조 로그를 구조 로거 호출로 치환하는 follow-up 커밋. 필드 누락이면 누락 필드 보강.

### R8.3 — trace_id 전파                                [근거: B8]
**트리거**: 서비스 간 호출 코드 작성.
**행동**: 요청 헤더에 `X-Trace-Id` 포함. 수신 측은 로그에 포함.
**검증**: 새 서비스 간 호출 코드에 trace 헤더 전파 확인.
**위반 시**: 신규 서비스 호출에 trace 헤더가 누락된 경우, 클라이언트·서버 양쪽을 모두 수정하는 follow-up. 기존 로그에서 trace_id가 없으면 해당 시점 관측이 재구성 불가임을 보고.

## R9 — 데이터 복구성

### R9.1 — 파괴적 마이그레이션은 백업 확인 선행            [근거: B9, NN6]
**트리거**: `DROP`, `TRUNCATE`, non-trivial `ALTER` 포함 마이그레이션.
**행동**: 백업 존재 + 복구 가능성 검증 후 실행. 복구 리허설 이력 확인.

### R9.2 — downgrade는 실제로 동작해야 한다              [근거: B9]
**트리거**: DB 마이그레이션 도구(예: Alembic, Flyway, Liquibase) 마이그레이션 작성.
**행동**: `downgrade()` 구현 + 스테이징에서 `upgrade → downgrade → upgrade` 테스트.
**금지**: 빈 downgrade 구현 (데이터 소실 불가피한 경우 제외, 이때는 명시적 주석 + 사용자 확인).
**검증**: PR에 리허설 결과 첨부.
**위반 시**: 빈 downgrade 또는 리허설 누락이 발견되면 해당 마이그레이션을 pending 상태로 되돌리고 보완 PR을 선행. 데이터 소실 불가피 사유는 주석으로 명시.

### R9.3 — 대용량 delete는 배치 처리                     [근거: B9, B8]
**트리거**: 대량 삭제/업데이트 스크립트 작성.
**행동**: `LIMIT` + 루프 + 진행 로그. 단일 트랜잭션으로 수만 행 이상 처리 금지.
**검증**: 스크립트에 `LIMIT` · 루프 · 진행 로그가 포함됐는지 코드 리뷰. 단일 트랜잭션 내 수만 행 처리 코드가 없는지 `rg "DELETE|UPDATE"` 로 점검.
**위반 시**: 단일 대량 트랜잭션을 발견하면 실행 금지. 배치 버전으로 재작성한 PR 제안.

### R10.2 — 외부 노출 엔드포인트는 trust-boundaries 등록    [근거: B10]
**트리거**: 새 공개 엔드포인트 추가.
**행동**: `harness/references/trust-boundaries.md`에 명시.

### R10.3 — 신뢰 경계 변경은 다이어그램 동기화           [근거: B10]
**트리거**: 인증/인가 로직 수정, 새 외부 노출 엔드포인트 추가.
**행동**: `harness/references/trust-boundaries.md`(또는 보안 경계 문서)에 반영 제안.
**금지**: 인증 로직 수정 + 문서 미반영 PR.
**검증**: 인증/인가/외부 노출 관련 diff가 있을 때 보안 경계 문서 변경도 같이 있는지 `git diff --name-only` 로 확인.
**위반 시**: 문서 미반영 PR은 머지 전 문서 업데이트 커밋을 선행. 이미 머지됐다면 follow-up PR을 즉시 생성.

### R10.4 — 시크릿 스캔                                  [근거: B10]
**트리거**: 모든 커밋 전.
**행동**: 커밋 예정 파일에 다음 패턴 검색:
```bash
rg -n "(api_key|secret|password|token|bearer)\s*[:=]\s*['\"][^'\"]{8,}['\"]" \
  --glob '!*.example' --glob '!*.md'
```
발견 시 작업 중단, 사용자 확인.
**검증**: pre-commit 시크릿 스캔 훅(프로젝트에서 구성)이 커밋 예정 파일을 스캔. 미구성이면 위 `rg` 명령을 수동 실행하고 0건이면 "clean" 확인.
**위반 시**: 스캔 없이 커밋된 이력이 발견되면 즉시 history scrub 또는 force-push 절차(사용자 승인 필수) 제안 + 관련 시크릿 로테이션.

## R11 — 문서 선행

### R11.1 — plan 없이 신규 기능/구조 변경 금지             [근거: B11, NN8]
**트리거**: 신규 엔드포인트, 스키마 변경, 리팩터링 중 구조 변경.
**행동**: 시작 전 `harness/plans/ongoing/<plan>.md` 작성. tracker.md 갱신.
**강제되지 않는 경우**: 오타·포맷 수정, 기존 기능 테스트 추가, 이미 plan에 포함된 하위 단계.

### R11.2 — plan이 필요 없는 작업 판정                   [근거: B11]
**트리거**: 작업 요청 수신 시 R11.1 대상인지 판단.
**행동**: "plan 생략 허용" 목록에 해당하는 작업만 plan 없이 진행. 그 외는 R11.1로 회귀.
**plan 생략 허용**:
- 오타, 포맷, 주석 수정
- 기존 기능에 대한 테스트 추가
- 이미 기존 plan에 포함된 작업의 하위 단계
- 명백한 버그 수정 (동작 변경 없음, 1-2파일 내)

**판정 불확실 시**: R0.1로 사용자에게 "이 작업은 plan이 필요한가요?" 질문.
**검증**: 작업 개시 전 허용 목록과 1:1 매칭되는지 자기 서술. 매칭이 애매하면 R0.1.
**위반 시**: 생략 판정이 잘못됐다고 판단되면 진행 중 작업을 중단하고 R11.1의 plan 경로로 재진입.

### R11.3 — tracker 갱신                                [근거: B11]
**트리거**: plan 기반 작업의 시작·중간 체크포인트·완료.
**행동**: 관련 `harness/plans/tracker.md` 에 상태 반영 초안 제시.
**금지**: tracker 갱신 없이 PR 머지 요청.
**검증**: PR 제출 전 tracker.md 관련 항목이 현재 작업 상태를 반영하는지 확인. diff 미포함 상태로 머지 요청 금지.
**위반 시**: tracker 미반영 머지가 발견되면 follow-up 커밋으로 즉시 갱신. 반복되면 R7.2 (a) 분류.

## R13 — Agent Teams (experimental)

플랫폼이 제공하는 multi-session agent team 기능 사용 규칙.
**선행 조건**: R0.6의 선택 트리에서 "Agent Team"으로 분기된 경우에만 적용. 경량 병렬은 R0.6의 Task/Explore 경로를 사용하고 이 카테고리로 오지 않는다.
**플랫폼 세부**: `harness/core/platforms/claude-code.md` §5 (플랫폼별 활성화·제약은 platforms 가이드에 위임). **파이프라인 배치**: `harness/core/workflows/pipeline.md` §6-bis.
**공식 문서 예시**: <https://code.claude.com/docs/en/agent-teams> (Claude Code 기준).

### R13.1 — Team spawn은 lead 세션에서만       [근거: B7]
**트리거**: team 생성 도구 호출(예: Claude Code `TeamCreate`) 또는 자연어 "팀 만들어줘" 요청.
**행동**: 현재 세션이 lead인지 확인. lead만 team을 spawn한다. teammate가 또 team을 만들 수 없다 (공식 문서 "no nested teams").
**검증**: spawn 전 "나는 lead인가?" 확인. teammate로 동작 중이면 lead에게 위임 메시지.
**위반 시**: teammate가 team 생성 도구를 호출하려 하면 즉시 중단, lead에게 알림.

### R13.2 — Team 크기 가이드                  [근거: B3, B12]
**트리거**: team 구성 설계 시.
**행동**:
- **3–5 teammates 권장** (공식 문서 best practices: "Start with 3-5 teammates for most workflows").
- **5–6 tasks per teammate 권장**. 15개 독립 태스크 → 3명이 출발점.
- **상한 없음**, 단 토큰 비용은 teammate 수에 선형 증가. B3 "지루한 기술 + 팀 적합성" 관점에서 과도한 확장 지양.
**검증**: spawn prompt에 명시된 teammate 수가 3–5 범위인지 확인. 초과하면 사유 명시.
**위반 시**: 사유 없이 6명+ 구성하면 advisor에 재확인.

### R13.3 — Risky 작업은 lead에서만 실행       [근거: B10, R10.x]
**트리거**: team 작업 중 파괴적/외부 영향 작업(다음 목록).
**행동**: 아래 작업은 teammate에서 실행 금지. lead가 수행한다.
- 인프라 apply/delete (예: `kubectl apply`, `terraform apply`, `docker stack deploy`, `secrets/` 포함 전반)
- `git push` / `git push --force` / `git reset --hard` / 브랜치 삭제
- 외부 서비스 전송 (이메일, Slack, API webhook, PR/이슈 생성)
- DB 마이그레이션 실행, 프로덕션 데이터 변경
- 의존성 설치·업그레이드 (`pip install`, `npm install -g`, `apt install`)
- `rm -rf` 및 유사 파괴적 쉘 명령
**근거**: teammate는 lead의 permission 모드를 spawn 시점에 상속하며, 이후 개별 모드 변경이 가능하지만 다수 teammate가 동시에 risky 작업을 수행하면 감사·롤백이 어렵다.
**검증**: team 작업 분해 시 risky step을 별도 "lead 전용 task"로 명시.
**위반 시**: 롤백 절차 준비 후 수동 확인. R10 체크리스트 전체 재실행.

### R13.4 — 세션 내 완결 원칙                 [근거: B7]
**트리거**: team 작업이 한 세션에 끝나지 않을 것으로 예상될 때.
**행동**: 세션을 넘기지 않는다.
- 플랫폼에 따라 session resume이 teammate를 복원하지 못할 수 있다 (예: Claude Code in-process 모드는 `/resume`으로 teammate 복원 불가). lead가 존재하지 않는 teammate에게 메시지를 보내는 상태가 된다.
- 세션 종료 전에 team cleanup (R13.5). 다음 세션에 재개가 필요하면 작업 산출물(파일, plan, tracker)로만 인수인계하고 team은 재-spawn.
**검증**: 세션 끝나기 전 `git status`로 teammate의 미커밋 작업 수렴 확인. team 미제거 상태로 세션 종료 금지.
**위반 시**: 세션 재개 후 lead가 이상 동작하면 team 상태 초기화 후 재-spawn.

### R13.5 — Cleanup 의무                      [근거: B7]
**트리거**: team 작업 완료, 또는 세션 종료 직전.
**행동**:
1. 모든 teammate를 shutdown (lead가 각각에 shutdown 메시지).
2. active teammate 없음 확인.
3. lead가 cleanup 요청. teammate 스스로 cleanup 금지 (공식 문서 Warning: 팀 컨텍스트 해석 오류 위험).
4. tracker/plan의 team 관련 상태 갱신.
**검증**: cleanup 완료 후 shared resources (task list 등) 잔존 여부 확인.
**위반 시**: orphaned teammate 또는 stale task list. lead 재시작으로 해결.

### R13.6 — 플랫폼별 병렬 스킬과 혼동 금지      [근거: B7]
**트리거**: 플러그인/확장이 제공하는 "parallel team" 또는 "agent picker" 스킬 호출 (예: Everything Claude Code의 `everything-claude-code:team-builder` 스킬).
**행동**: 다음 두 메커니즘을 구별한다.
- **플러그인 스킬** (예: ECC `team-builder`, if installed) → 내부적으로 `Task(subagent_type=...)` multi-tool 병렬 구성. **R0.6 선택 트리의 "Task 병렬"** 경로.
- **플랫폼 native agent teams** (예: Claude Code `TeamCreate` + teammates) → R13 전체가 적용되는 **별도 세션 기반** 경로.
**검증**: 사용자가 "team" 언급 시 어느 메커니즘을 의도하는지 물어본다 (R0.1 불확실하면 물어본다).
**위반 시**: 잘못 선택하면 R0.6 선택 트리를 역방향으로 재설계.
