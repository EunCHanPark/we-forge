## B1 — 의존성은 단방향

**원칙**: 서비스 간 런타임 호출은 항상 한 방향으로만 흐른다.

```
hooks → events.jsonl → pattern-detector → ECC dedup → synthesize → ledger.jsonl
```

**위반 신호**:
- 하위 레이어가 상위 레이어 URL/호스트명 hardcoding
- DB 트리거가 외부 HTTP 엔드포인트 호출
- 프론트엔드가 DB write를 직접 수행

**B6와의 관계**: B1이 *런타임 호출 방향*을 규정한다면, B6는 *코드 소유권과 책임 분할*을 규정한다.

---

## B2 — 레포가 유일한 진실의 원천

**원칙**: 인프라, 스키마, 설정은 모두 레포 안에 코드로 존재한다. 서버 수동 변경은 반드시 레포에 반영한다.

**적용**:
- DB 스키마 변경 → 마이그레이션 파일로 커밋
- 배포 리소스 변경 → 매니페스트에 반영
- 환경 설정 추가 → `.env.example`에 키 추가 (값 제외, 자세한 패턴은 B5)

**예외 절차 (hotfix)**:
- 프로덕션 장애로 서버 수동 변경 시: **24시간 내** 레포 반영 PR 필수
- PR 제목에 `[hotfix-reconcile]` 태그, 본문에 변경 이력 기록

**절대 예외**:
- `secrets/`는 실제 값을 커밋하지 않는다. PLACEHOLDER만 유지하고 서버에서 수동 적용.

---

## B3 — 지루한 기술 + 팀 적합성

**원칙**: 기술 선택은 두 기준을 모두 통과해야 한다. (1) 검증되어 놀라움이 적을 것. (2) 우리 팀이 2년 후에도 유지보수 가능할 것.

**왜**: CV1(간결성)의 기술 스택 적용. 새 의존성은 운영 복잡도·학습 비용·공급망 리스크를 모두 늘린다. NN7이 절차적으로 이 원칙을 강제한다.

**새 런타임 의존성 추가 절차**: 현재 선언된 패키지 매니페스트에 없는 라이브러리는 사용자 승인을 거친다. 에이전트는 제안만 가능.

---

## B4 — 테스트 없으면 완료가 아니다

**원칙**: 신규/변경 코드는 테스트와 함께 커밋한다. PR은 커버리지를 떨어뜨리지 않는다.

**측정 가능한 기준**:
- **신규 API 엔드포인트**: 최소 1개의 통합 테스트 (성공 + 인증 실패)
- **신규 프론트엔드 컴포넌트**: 상태 변화가 있는 경우 테스트
- **DB 마이그레이션**: 스테이징에서 up/down 모두 검증 후 프로덕션 적용
- **CI 강제**: 커버리지 하락 시 PR 머지 차단 (프로젝트 정책에 따라)

**Mock 사용 원칙**: 외부 의존성(네트워크, 파일시스템)만 mock한다. 비즈니스 로직 전체를 mock으로 대체한 테스트는 PR 거부 사유.

---

## B5 — 환경 설정은 코드로

**원칙**: 서비스가 동작하는 데 필요한 설정은 레포에 코드로 표현된다.

**적용**:
- 모든 서비스 디렉토리에 `.env.example` 유지 (키 목록, 한 줄 설명, 기대 형식 포함)
- 설정 매니페스트(Docker Compose / ConfigMap)로 관리
- "서버에서 직접 고쳤다"는 사고방식 금지 (→ B2 hotfix 절차)

---

## B6 — 모듈 경계는 명확하다

**원칙**: 각 서비스/디렉토리는 단일 책임을 갖는다. 경계를 넘는 참조는 정의된 API로만.

**B1과의 관계**: B1은 호출 *방향*을, B6는 호출 *인터페이스*를 규정한다. 방향이 맞아도 경계를 우회하면 B6 위반.

---

## B7 — 에이전트 실수는 하네스로 고친다

**원칙**: 에이전트가 실수를 반복한다면 **코드를 고치는 게 아니라** 하네스(가이드라인, 훅, 결정 기록)를 고친다.

**단, 층위를 구분한다**:
- **하네스 공백에서 온 실수** → 하네스 수정 (규칙 추가)
- **코드 구조에서 온 실수** (모호한 명명, 흩어진 책임 등) → 코드 수정
- **모델 능력 한계에서 온 실수** → 작업 분할·우회

**중요한 제약 (NN5)**: 하네스 자체(`CLAUDE.md`, `harness/*`, `harness/*`)는 **사용자 통제 영역**이다. 에이전트는 이를 자율 수정하지 않는다.

---

## B8 — 시스템은 관측 가능해야 한다

**원칙**: 장애가 났을 때 원인을 찾을 수 있어야 한다.

**적용**:
- **구조화 로그 필수**: JSON 형식, 최소 `timestamp`, `level`, `service`, `trace_id`, `user_id`(가능한 경우)
- **메트릭**: 핵심 엔드포인트는 지연/에러율/처리량 수집
- **분산 추적**: 서비스 경계는 `trace_id` 전파
- **에러는 삼키지 않는다**: 예외 처리 시 반드시 로그 + 알림 분기. `except: pass` 금지.

---

## B9 — 데이터는 복구 가능해야 한다

**원칙**: 잃으면 안 되는 데이터는 정의된 RPO/RTO를 갖는다.

**적용**:
- **백업 정책 분류**:
  - Tier 1 (사용자 데이터, 인증): 일일 백업, RPO 24h, RTO 2h
  - Tier 2 (설정, 감사 로그): 주간 백업, RPO 7d, RTO 24h
  - Tier 3 (캐시, 파생 데이터): 백업 불필요, 재생성 경로 문서화
- **복구 훈련**: 분기 1회 스테이징에서 백업 복원 리허설
- **파괴적 마이그레이션 절차**: `DROP`/`TRUNCATE`/non-trivial `ALTER`는 downgrade가 작동해야 함

**NN6의 진짜 충족 조건**: 백업이 존재한다는 것과 그 백업으로 실제 복구가 된다는 것은 다르다. 복구 리허설이 없으면 NN6 충족은 서류상에 그친다.

---

## B10 — 신뢰 경계는 명시적이다

**원칙**: 누가 누구를 어떤 근거로 신뢰하는지 그림으로 존재해야 한다.

**적용**:
- **신뢰 경계 다이어그램** (`harness/references/trust-boundaries.md`):
  - 공개 인터넷 → Ingress → 서비스
  - 서비스 간 인증 방식 (JWT? mTLS? 공유 시크릿?)
  - DB 접근 권한 (RLS? 역할 기반? 서비스 계정?)
- **최소 권한 원칙**: 서비스 계정은 필요한 최소 권한만
- **인증은 경계에서**: 내부 서비스는 경계가 인증을 통과시킨 요청을 신뢰. 단, `trace_id`와 `user_id`는 전파 검증.

---

## B11 — 문서가 구현에 선행한다

**원칙**: 새 기능·구조 변경·스키마 변경은 **plan 문서가 먼저 존재**해야 시작할 수 있다.

**왜**: NN8("문서 없이 기능을 추가하지 않는다")은 Non-Negotiable이다. 문서가 없으면 의도(spec)가 머릿속에만 존재하고, 구현이 그 의도를 충족하는지 검증할 기준도 없다.

**적용**:
- **선행 문서 위치**: `harness/plans/`
- **plan 문서 최소 요건**: 목적, 범위, 완료 기준, 관련 원칙 링크
- **tracker.md 갱신**: plan 시작/진행/완료 상태 반영 (DoD #5)

**강제되지 않는 경우**: 오타·포맷·주석 수정, 기존 기능에 대한 테스트 추가, 이미 plan에 포함된 작업의 하위 단계.

**강제되는 경우**: 신규 엔드포인트·화면·워크플로, 스키마 변경, 리팩터링 중 구조가 바뀌는 작업, 새 외부 의존성 추가.

---

## B12 — 작고 분리된 변경

**원칙**: 변경은 작게 쪼개 자주 낸다. **구조 변경과 기능 변경은 한 PR에 섞지 않는다.**

**왜**: CV5/CV6의 통합. 큰 PR은 리뷰 품질을 떨어뜨리고 롤백 단위를 뭉뚱그린다. 리팩터링과 기능 추가가 섞인 PR은 "이 버그가 리팩터링에서 왔는가 새 기능에서 왔는가"를 사후 판별 불가능하게 만든다. Kent Beck의 "Tidy First" 원칙과 같은 뿌리다.

**PR 분할 기준**:
- **구조 변경(Structural)**: 동작을 바꾸지 않고 코드 형태만 바꾸는 변경. 이름 변경, 파일 이동, 함수 추출, 중복 제거.
- **기능 변경(Behavioral)**: 외부에서 관찰 가능한 동작이 바뀌는 변경. 새 엔드포인트, 로직 수정, 버그 수정.
- **두 종류가 필요하면 PR을 나눈다.** 구조 먼저 → 머지 → 기능 PR.

**PR 크기 가이드**:
- 권장: diff 300라인 이하, 파일 10개 이하
- 300~800라인: PR 본문에 분할이 왜 어려운지 설명
- 800라인 초과: 분할 재검토 (단, 자동 생성 파일·마이그레이션 제외)

**위반 신호**:
- "리팩터링 + 신규 기능 + 버그 수정"이 한 PR에
- PR 본문에 "이것도 김에 고쳤습니다" 류 표현
- 리뷰어가 "어느 변경이 어느 목적인지 모르겠다"고 피드백
