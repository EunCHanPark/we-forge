# Repository Reliability

## 1. 에러 처리
- 에러를 삼키지 않는다. 빈 catch/except 블록 금지.
- 에러에 컨텍스트를 포함한다 (무엇이, 어디서, 왜 실패했는지).
- 비즈니스 예외와 시스템 예외를 구분한다.

## 2. 로깅
- 구조화 로그를 사용한다 (timestamp, level, service, trace_id).
- 로그 레벨: ERROR(즉시 확인), WARN(잠재 문제), INFO(주요 흐름), DEBUG(개발 전용).
- 프로덕션에서 DEBUG 비활성화.
- 앱별 로그 필드와 알림 규칙은 app-level 문서에 명시한다.

## 3. 재시도
- 일시적 장애(네트워크, 연결 풀)는 지수 백오프로 재시도한다.
- 인증 실패(401), 권한 없음(403), 검증 실패(400/422)는 재시도하지 않는다.
- 최대 재시도 횟수를 정한다. 무한 재시도 금지.

## 4. 타임아웃
- 모든 외부 호출에 타임아웃을 설정한다. 기본값 없이 열어두지 않는다.
- 앱별 구체적 타임아웃 값은 app-level 문서에 명시한다.

## 5. 그레이스풀 디그레이데이션
- 부분 장애가 전체 서비스 장애로 이어지지 않게 한다.
- 장애 발생 시 사용자에게 상태를 알린다.
- 모든 프로덕션 서비스는 health check 엔드포인트를 제공한다.

## 6. 백업·복구
- 프로덕션 데이터는 백업과 롤백이 정의되어야 한다 (Core Value 4).
- 앱별 백업 주기, RPO/RTO는 app-level 문서에 명시한다.
